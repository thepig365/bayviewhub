# Bayview Landing Page SEO / AEO / GEO Operating Playbook

_Last updated: 2026-03-29_

## What this document is for

Use this playbook for any Bayview landing page that is expected to:

- rank for a narrow topic
- answer clearly enough to be quoted or summarized
- convert traffic into an identifiable next step
- feed a reporting and review loop

This is not a “write nicer copy” checklist. It is an operating standard.

---

## The search reality in 2026

A strong landing page now has to satisfy four systems at once:

1. **Traditional search** — crawlable, canonical, internally linked, and topically clear.
2. **Answer engines** — easy to extract, with question-led sections and concise factual answers.
3. **AI-assisted search surfaces** — clear definitions, boundaries, and source-backed claims.
4. **Measurement systems** — traffic, engagement, source, and action tracking must exist before promotion.

### Non-negotiable implications

- Do not overclaim in titles or hero copy if the body is more qualified.
- Do not rely on FAQ schema alone for visibility. Google still supports `FAQPage` structured data, but FAQ rich results are now shown primarily for well-known government and health sites, so the value is mostly **understanding**, not guaranteed enhanced SERP treatment.
- Structured data helps search engines understand the page, but it does not guarantee a rich result.
- Canonicalization, sitemap inclusion, and internal linking should all point to the same preferred URL.
- AI features in Google Search use the same technical foundations as Search. Inclusion is not opt-in magic; it depends on crawlability, renderability, quality, and content controls.

---

## Definitions

### SEO
Make the page discoverable, indexable, canonical, and topically obvious.

### AEO
Make the page directly answer likely user questions in a format that can be lifted into summaries, snippets, and answer surfaces.

### GEO
Make the page safe and useful to cite: explicit definitions, boundaries, source-backed claims, and enough context that an AI system is less likely to distort it.

### Campaign operations
A landing page is not complete until traffic, engagement, CTA clicks, and review cycles are measurable.

---

## Core principle

**One landing page = one primary topic + one primary CTA + one operating pack.**

Support pages can carry secondary intents, but the main landing page must have one job.

## The highest-achievable strategy for Bayview

## 1. Build every landing page around a single canonical topic

For each page, define:

- **Primary topic** — one thing only
- **Canonical URL**
- **Primary CTA**
- **3 support angles max**

Example pattern:

- Primary topic: `Victorian backyard small second home / SSD rules + feasibility`
- Primary CTA: `Open the feasibility checklist`
- Support angles:
  - rules and permit distinction
  - cost and value context
  - use cases / house types

### Why this matters
If the topic is broad, Google and AI systems get ambiguity. If the CTA is broad, users drift. If the support angles are too many, the page becomes a directory instead of an answer asset.

---

## 2. Use a proof-first page architecture

### Default page structure

1. Hero
2. Quick Answer
3. Short framing
4. Visual product / archetype section (when the topic is spatial, visual, or physical)
5. Pathway / programme map
6. How it works
7. FAQ
8. Primary CTA
9. Evidence / source block where relevant

### Rules
- Keep the **hero** short.
- Put the **best answer** near the top, not buried mid-page.
- Use visual mid-page anchors for property, design, or product pages.
- Push deeper edge cases into support pages instead of bloating the hub page.

---

## 3. Write answer blocks that are easy to extract

Every important landing page should include:

### A. Standard definition sentence
One sentence that defines the thing precisely.

### B. Three factual bullets
These are the non-negotiable facts the page can defend.

### C. One risk / boundary sentence
This is where overclaiming gets shut down.

### D. One positioning sentence
What Bayview is actually doing — clearly and without pretending to be something else.

### Example pattern
- Definition: what it is
- Facts: what is true most of the time
- Boundary: what still needs checking / what this page does **not** decide
- Positioning: how Bayview uses the information

### Writing rules
- Prefer plain English over legalistic padding.
- Prefer conditions over slogans.
- Prefer short paragraphs over long blended prose.
- Avoid duplicate restatements of the same facts in the hero, quick answer, FAQ, and CTA bands.

---

## 4. Keep FAQ, but use it correctly

### Use FAQ for:
- real user-language questions
- concise answers
- clarifying common misunderstandings
- giving answer engines highly extractable blocks

### Do not use FAQ for:
- repeating the hero in question form
- padding thin content
- pretending FAQ rich results are the main outcome

### FAQ structured data rule
Keep `FAQPage` JSON-LD if the page genuinely contains a clear FAQ list. Treat it as a **content-understanding layer**, not a guaranteed SERP feature.

---

## 5. Make every claim auditable

For each landing page, classify claims into four buckets:

### Strong claims
Source-backed and defensible.

### Framing claims
Positioning or interpretation, not direct facts.

### Risky claims
Likely to be misunderstood or over-broadened.

### Decorative claims
Not harmful, but not useful.

### Operating rule
Delete or soften risky claims before promotion. Decorative claims should not crowd out strong claims.

---

## 6. Use structured data for understanding, not wishful thinking

### Recommended baseline
- `Organization`
- `WebSite`
- page-level type that genuinely matches the page
- `FAQPage` if the page contains a real FAQ
- other schema only if it truthfully reflects the page

### Rules
- Do not add schema just because a plugin allows it.
- Visible content and JSON-LD should align.
- Keep the same facts, names, and boundaries in both places.

---

## 7. Canonical, sitemap, and internal linking must agree

### Required standard
- one canonical URL per landing page
- that URL included in sitemap
- internal links should point to the canonical URL
- redirects should reinforce the preferred version where duplicates exist

### Why
Google treats redirects and `rel="canonical"` as strong signals, and sitemap inclusion as a weaker reinforcing signal. These signals work better together than separately.

---

## 8. AI-search readiness is mostly content clarity, not hacks

### Practical GEO rules
- define the subject early
- name the constraints explicitly
- distinguish facts from estimates
- show the source path
- state what still requires verification
- remove noisy sales language that weakens trust

### Avoid
- inflated certainty
- “best”, “guaranteed”, “always”, “no permit” style claims unless absolutely true in all cases
- fashionable AI-discovery gimmicks with no evidence of impact

### Treat optional items carefully
Files like `llms.txt` can be kept as a supplemental discovery aid, but they are **not** a substitute for crawlable, well-structured page content and should not be the core of the strategy.

---

## 9. Instrument the page before promotion

### Minimum event set
Track at least:

- page views
- source / medium / campaign / referrer
- average engagement time / engaged sessions
- scroll depth
- key CTA clicks
- support-page clicks
- form starts
- form submits
- share clicks

### Why
A landing page that cannot tell you:
- who came
- from where
- how long they stayed
- what they clicked
is not operationally complete.

---

## 10. Alerting should be useful, not noisy

### Do not do
One email per page view.

### Do
Use a three-level model:

#### Instant alerts
- first traffic after publish or quiet period
- new traffic source
- traffic spike
- key CTA spike
- form submit

#### Daily digest
- views
- users / sessions if available
- source breakdown
- top referrers
- average engagement time
- scroll depth summary
- CTA clicks
- form starts / submits

#### Weekly digest
- source trend
- top channel
- engagement trend
- conversion trend
- AI-search test notes
- content update needs

---

## 11. Use a campaign operating pack for every important page

Each landing page should have a markdown operating pack with:

- standard definition sentence
- three factual bullets
- one risk / boundary sentence
- one positioning sentence
- primary topic
- canonical URL
- main CTA
- three promotion angles
- claim audit summary
- four fixed LLM test prompts
- 7 / 14 / 30 review criteria
- scorecard:
  - Brand Citation
  - Topic Ownership
  - Proof & Trust
  - Distribution & Feedback

This turns a page into a repeatable campaign asset.

---

## 12. Use a 7 / 14 / 30 review cycle

### 7-day review
- page indexed?
- canonical correct?
- internal links used?
- first traffic sources visible?

### 14-day review
- social traffic quality
- engagement time
- CTA click-through
- first AI-search prompt checks

### 30-day review
- search impressions / clicks
- top queries
- AI-search mention accuracy
- copy too heavy / too weak?
- update `Last updated` if the topic changed materially

## 13. A practical Bayview implementation standard

For Bayview landing pages, the default standard should be:

### Tone
- proof-first
- grounded
- non-overclaiming
- not generic builder hype
- not faux-luxury filler
- not AI-slop

### Content style
- clear definition first
- question-led headings where useful
- image-led mid-page section if the topic is spatial or product-like
- FAQ for real doubts
- explicit evidence / source path for regulated topics

### Promotion style
- landing page first
- support pages for rule, cost, fit, and edge-case depth
- campaign pack created before scaling distribution
- analytics and alerting installed before major promotion

## 14. Priority implementation checklist

### Page hardening
- [ ] primary topic defined
- [ ] canonical URL set
- [ ] title and meta aligned with real claims
- [ ] H1/H2 structure clean
- [ ] Quick Answer present
- [ ] FAQ present and non-duplicative

- [ ] strong claims identified
- [ ] risky claims softened
- [ ] visible copy matches structured data

### Measurement

- [ ] attribution tracked
- [ ] engagement tracked
- [ ] CTA clicks tracked
- [ ] form events tracked
